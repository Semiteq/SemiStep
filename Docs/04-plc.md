## 4.1. Общая концепция

- Конфигурация структуры рецепта хранится на стороне PC (YAML-файлы).
- Обмен данными выполняется по протоколу S7 (библиотека S7.Net).
- На стороне PLC выделяются отдельные датаблоки (DB) для разных типов данных.

---

## 4.2. Протокол и транспорт

| Параметр | Значение |
|----------|----------|
| Протокол | S7 (Siemens) |
| Библиотека | S7.Net |
| PLC | Siemens S7-1200 / S7-1500 |
| Порт | 102 (по умолчанию) |

### Настройки подключения

| Параметр | Описание |
|----------|----------|
| IP-адрес | Адрес ПЛК |
| Rack | Номер стойки |
| Slot | Номер слота CPU |
| Header DB | Номер датаблока Header |

## 4.3. Структура данных в PLC

Данные разделены по датаблокам:

| Датаблок | Назначение |
|----------|------------|
| DB_HEADER | Служебная информация, номера других DB |
| DB_MANAGING | Координация транзакций записи |
| DB_INT | Массив целочисленных данных рецепта |
| DB_FLOAT | Массив вещественных данных рецепта |
| DB_STRING | Массив строковых данных рецепта |
| DB_EXECUTION | Состояние выполнения рецепта |

### Header (DB_HEADER)

Читается при инициализации. Содержит:

- Версию протокола
- Номера датаблоков для остальных областей
- Служебные флаги

### Managing Area (DB_MANAGING)

Используется для атомарных операций записи:

- Статус транзакции (PC и PLC)
- ID транзакции
- Контрольные суммы данных
- Коды ошибок

---

## 4.4. Операции обмена

### 4.4.1. Чтение рецепта

Приложение считывает:

- Текущее количество строк рецепта
- Значения параметров из DB_INT, DB_FLOAT, DB_STRING
- Состояние валидности (по контрольным суммам)

### 4.4.2. Запись рецепта

**Ключевое требование:**

> Запись выполняется **атомарно** с подтверждением через контрольные суммы.

Механизм:

1. PC устанавливает статус "writing" и записывает данные
2. PC запрашивает commit
3. PLC вычисляет контрольные суммы и сравнивает
4. При совпадении — транзакция успешна
5. При ошибке — повторная попытка (до 3 раз)

### 4.4.3. Обработка ошибок

| Ошибка | Действие |
|--------|----------|
| Checksum mismatch | Повтор транзакции (макс. 3 раза) |
| Timeout | Проверка связи, повтор или сообщение пользователю |
| PLC busy | Ожидание или сообщение "ПЛК занят" |

## 4.5. Мониторинг выполнения

Приложение периодически читает DB_EXECUTION:

| Данные | Назначение |
|--------|------------|
| `recipe_active` | Блокировка редактирования при выполнении |
| `actual_line` | Подсветка текущего шага в таблице |
| `step_current_time` | Расчёт оставшегося времени |
| `for_loop_count_1/2/3` | Отображение итераций вложенных циклов |

### Режимы UI

- **recipe_active = true** → режим мониторинга (редактирование заблокировано)
- **recipe_active = false** → режим редактирования

### Расчёт времени

```
LineTimeLeft = max(0, Длительность_шага − step_current_time)
TotalTimeLeft = max(0, Общая_длительность − (Время_начала_шага + step_current_time))
```

---

## 4.6. Обработка ошибок связи

При ошибках подключения/обмена:

- Отображать пользователю ясные сообщения о сбоях
- Предотвращать переход приложения в неконсистентное состояние
- Давать возможность повторить операцию без перезапуска приложения
